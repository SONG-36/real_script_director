# 1. 文档目的

本文件用于解决两个容易混淆的问题：

1. 从长期来看，一个真正完整的 TikTok 商品内容系统应该包含哪些决策和生产模块；
2. 当前正在建设的 Script Director Custom GPT 应严格收敛在哪个环节。

本文件不要求现在一次性开发完整系统。

它的作用是提供一张长期地图，避免未来出现两种错误：

- 只建设脚本生成器，却遗漏“商品为什么适合这种内容”的上游判断；
- 为了补足上游判断，把所有功能继续塞入同一个 Custom GPT，导致职责模糊、指令冲突和输出不稳定。

核心结论：

> 完整系统可以很大，但每一个运行模块必须足够小、职责单一、输入输出明确。

回答：

```text

```

这部分属于：

```text
SD-1｜Script Director
当前 Custom GPT
```

两个系统必须连接，但不能混为一个 Agent。

这是本系统最重要的边界。

```text

```

```text

```

当前 Custom GPT 不负责。

## 4.2 第二层：产品级内容组合决策

发生在单条视频构想之前。

回答：

```text
一个商品第一轮需要哪几类内容？

哪些由团队自产？
哪些交给达人？
哪些暂时不做？
```

输出示例：

```yaml
产品内容组合:
  商品: 电动泡沫喷壶
  第一轮内容:
    - 类型: 产品证明
      数量: 3
    - 类型: 痛点对比
      数量: 2
    - 类型: 使用场景
      数量: 2
  自产内容: 产品证明、痛点对比、使用场景
  达人内容: 暂不作为第一轮核心
  核心验证:
    - 免反复按压是否具备明显购买说服力
    - 连续喷射是否能在小屏画面中被清楚理解
```

责任模块：

```text
CP-0｜Content Portfolio Planning
```

当前 Custom GPT 不负责。

## 4.3 第三层：单条构想内部的脚本策略

发生在构想已经确定以后。

回答：

```text

适合什么内容形式和结构？

画面怎么证明？
每一秒怎么拍？
```


# 5. SD-0｜内容方向决策系统

## 5.1 目的

在投入脚本、拍摄和剪辑成本前，判断：

```text
是否应该为该商品建设某种内容路线。
```

它不是爆款预测系统，也不能保证商业结果。

它只负责把市场观察、产品特征、账号资源和验证成本整理成可测试决策。

## 5.2 销售机制分类

每个商品可能主要依赖一种或多种机制。

| 销售机制 | 核心成交原因 | 常见内容路线 |
|---|---|---|
| 产品证明驱动 | 商品能力可快速、直观地被画面证明 | 实拍演示、连续 Proof、对比 |
| 痛点场景驱动 | 用户能够立即认出问题和使用场景 | 痛点开场、问题解决、场景种草 |
| 达人信任驱动 | 用户更依赖创作者人格、经验和背书 | 达人带货、真人体验、专业测评 |
| 教育解释驱动 | 用户需要先理解用途、差异或使用方式 | 教程、解释、测评、问答 |
| 价格优惠驱动 | 同质化较高，主要购买理由是价格或优惠 | Offer、直播、促销、达人优惠 |
| 情绪身份驱动 | 商品承载审美、生活方式或身份认同 | 故事、生活方式、情绪内容 |
| 品牌信任驱动 | 成交来自已有品牌认知和安全感 | 品牌展示、用户证言、权威说明 |
| 混合驱动 | 两种以上机制共同决定成交 | 产品 Proof + 真人体验等 |

注意：

> 发现大博主爆款较多，只能说明需要调查“达人影响力是否是关键变量”，不能直接证明产品不适合产品内容。

## 5.3 产品—内容适配判断

### 维度一：痛点是否可视化

- 用户问题能否在前三秒被拍出来；
- 问题是否常见；
- 观众是否无需大量解释就能理解。

### 维度二：产品动作是否可视化

- 产品工作过程能否连续展示；
- 关键动作是否能在手机画面中看清；
- 是否需要大量口播才能解释。

### 维度三：结果是否可视化

- 前后差异是否明显；
- 能否避免跳剪和夸大；
- 结果是否真正由商品产生。

### 维度四：差异是否可视化

- 与传统方案或竞品的区别能否被直观看见；
- 差异是否需要参数解释；
- 差异是否具有购买意义。

### 维度五：信任门槛

- 用户是否必须相信某个人后才愿意购买；
- 是否涉及长期效果、健康、美容、安全或主观体验；
- 官方号纯产品演示能否建立足够可信度。

## 5.4 发布主体适配

路线决策必须区分：

```text
产品适不适合做内容
```

和：

```text
我们的账号适不适合承接这种内容
```

发布主体包括：

| 发布主体 | 更适合的作用 |
|---|---|
| 官方号 | 产品事实、Product Proof、品牌与功能解释 |
| 渠道号 | 场景测试、痛点内容、素材验证 |
| 普通 UGC 创作者 | 生活体验、自然口播、弱商业感 |
| 头部达人 | 强曝光、人格信任和消费带动 |
| 垂类专业达人 | 测评、教育、专业解释 |
| 直播间 | 价格、优惠、即时演示和答疑 |

一个商品依赖达人，不等于产品本身完全不适合视频。

它可能只是：

```text
不适合由弱信任白牌官方号单独完成转化。
```

## 5.5 市场信号归因

研究爆款视频时，至少区分：

```text
创作者规模
历史平均播放
视频自然增量
是否投流
是否有价格优惠
是否品牌自带认知
商品是否已是成熟爆品
视频是否真正展示产品能力
评论是否围绕产品购买理由
```

禁止直接使用以下错误归因：

```text
大达人视频很火
→ 这个脚本一定好

大达人能卖
→ 小号复制脚本也能卖

小号没有爆款
→ 产品一定不适合内容

产品视频播放低
→ 达人路线一定更好
```

```yaml

  商品: ""

  市场观察:
    爆款主要来自: ""
    小中账号是否有成功案例: ""
    主要内容形式: ""
    是否明显依赖达人信任: ""
    是否明显依赖价格或优惠: ""
    观察局限: ""

  产品内容适配:
    用户痛点是否可视化: ""
    产品动作是否可连续展示: ""
    结果是否可直观证明: ""
    差异化是否能用画面表达: ""
    信任门槛: "低 / 中 / 高"

  当前资源:
    可用发布主体: []
    可用达人资源: ""
    可拍场景: []
    拍摄能力: ""
    可接受测试成本: ""

  决策:
    主要销售机制: ""
    主内容路线: ""
    发布主体: ""
    不推荐路线: ""
    本轮验证假设: []
    方向状态: "GO / TEST / NO-GO"
```

## 5.7 GO / TEST / NO-GO

### GO

有足够依据进入构想和脚本阶段。

### TEST

方向尚不确定，但值得用低成本、少数量视频验证。

### NO-GO

在当前产品条件、账号资源或拍摄能力下，不建议进入该内容路线。

`NO-GO` 不一定表示放弃商品，也可能表示：

```text
转达人合作
转直播
转价格路线
调整发布主体
等待商品改款
降低自产视频优先级
```

## 5.8 退出机制

出现以下情况时，应暂停或更换路线：

- 主要卖点无法形成真实 Product Proof；
- 脚本必须依赖夸张或不可证明 Claim；
- 拍摄成本长期高于测试价值；
- 商品高度同质化且没有价格、品牌或信任优势；
- 连续测试后无法确认用户对主要购买理由有感知；
- 仅头部达人有效，当前团队又无法获得对应达人资源；
- 实际效果与供应商承诺明显冲突；
- 内容表现问题无法通过 Hook、Proof、表达或拍摄改进解释。

退出规则不能只看单条视频低播放。

# 7. IC-0｜自有视频构想卡

构想卡代表：

> 已经决定值得进入脚本阶段的一条完整视频构想。

构想卡不应重新承担市场分析，也不需要填写主传播命题、核心卖点或脚本结构。

构想卡只回答：

```text
这条视频准备讲什么？
用户遇到什么问题？
产品怎么解决？
准备用什么画面证明？
现在有哪些拍摄条件和缺口？
```

构想卡是 SD-0 / CP-0 与 SD-1 之间的接口。

# 8. SD-1｜当前 Script Director Custom GPT

## 8.1 唯一职责

> 将用户已经确定的一条自有视频构想，结合已验证商品事实、待验证内容、风险限制和参考视频，转换成一份真实、可拍、可审核的 TikTok Shop US 正式拍摄脚本。

当前 Custom GPT 不是：

- 选品系统；
- 市场研究系统；
- 爆款分析系统；
- 达人策略系统；
- 账号运营系统；
- 产品内容路线决策系统；
- 产品级内容排期系统；
- 发布和投流系统。

## 8.2 输入

固定使用已批准的人工扁平 Script Input Pack：

```yaml
视频构想:
  视频主题: ""
  商品: ""
  视频目标: "转化"
  市场: "TikTok Shop US"
  用户痛点: ""
  产品怎么解决: ""
  画面怎么证明: ""
  拍摄条件: ""
  目前缺什么: ""

已验证商品事实:
  - ""

待验证内容: []

风险限制: []

参考视频: []
```

上游默认前提：

```text
人已经决定这条构想值得进入脚本阶段。
```

## 8.3 允许做的判断

Custom GPT 可以在给定构想内部判断：

- 输入是否足以支持脚本；
- 哪些内容是已验证事实；
- 哪些内容待验证；
- 生成状态是 READY、PROVISIONAL 还是 BLOCKED；
- 如何规范化口语构想；
- 主传播命题是什么；
- 核心卖点和辅助卖点如何组织；
- 这条构想适合什么内容形式；
- 采用什么脚本结构；
- Hook 如何设计；
- Product Proof 如何安排；
- 目标时长如何分配；
- 是否需要真人、手部操作或旁白；
- 每一个镜头如何执行；
- 脚本是否真实、清晰、可拍和合规。

这些判断全部发生在：

```text
已经给定的单条视频构想内部。
```

## 8.4 禁止做的判断

Custom GPT 不得：

- 判断商品是否值得进入市场；
- 判断产品是否应该上架；
- 决定商品整体应该走达人路线还是产品路线；
- 搜索市场、销量、达人或竞品数据；
- 分析“大达人带货是否证明该品依赖达人”；
- 决定官方号、渠道号和达人资源的总体分工；
- 设计整个产品第一轮内容组合；
- 自动生成几十条构想；
- 决定价格、优惠、投流和直播策略；
- 做账号定位和发布排期；
- 预测播放、销量、GMV 或转化率；
- 因为参考视频表现好就推断其脚本机制有效；
- 修改上游已经确定的商业路线。

## 8.5 遇到越界请求时的行为

### 用户要求市场判断

例如：

```text
这个商品适合达人带货还是产品视频？
```

Custom GPT 应回答：

```text
该问题属于上游内容路线决策，不属于本 Script Director。
请提供已经确定的视频构想和 Script Input Pack。
```

### 用户要求选品

例如：

```text
这个商品值不值得做？
```

应标记为范围外，不开展选品分析。

### 用户只提供商品资料，没有构想

应要求补充视频构想，而不是自动完成市场分析后替用户确定路线。

### 用户提供已确定构想

正常进入事实检查、策略设计和脚本生成。

## 8.6 固定输出

```text
1. 生成状态
2. 阻塞项和风险
3. 规范化构想
4. 视频策略
5. 主传播命题
6. 核心问题、核心卖点和辅助卖点
7. 逐镜头正式拍摄脚本
8. 英文字幕或旁白及中文审核含义
9. 拍摄准备清单
10. 两个 Hook 变体
11. 脚本审核结果
```

## 8.7 当前 Custom GPT 的成功标准

- 不改变用户核心构想；
- 不编造产品能力；
- 不把待验证内容写成确定结果；
- 不把竞品或 AI 视频当成我方 Product Proof；
- 不需要人工重写整份脚本；
- 团队能够直接理解和拍摄；
- 输出结构稳定；
- 能正确标记 READY、PROVISIONAL 和 BLOCKED；
- 不主动扩展到市场判断和内容路线决策。

# 9. 后续生产模块

以下模块属于完整系统，但不属于当前 Custom GPT。

## 9.1 PP-1｜拍摄准备

负责：

- 素材需求；
- 道具；
- 场景；
- 人员；
- 测试任务；
- 拍摄日程；
- 镜头连续性检查。

## 9.2 FF-1｜首帧

负责：

- 首帧构图；
- 商品真实性；

# 10. 双重状态体系

不同阶段不能共用一套状态。

## 10.1 路线状态

由 SD-0 输出：

```text
GO
TEST
NO-GO
```

回答：

```text
是否进入这种内容路线。
```

## 10.2 脚本状态

由 SD-1 输出：

```text
READY
PROVISIONAL
BLOCKED
```

回答：

```text
这条既定构想能否形成正式脚本。
```

错误示例：

```text
脚本事实充分，因此认为产品路线正确。
```

这是把两套状态错误合并。

可能出现：

```text
路线状态 = TEST
脚本状态 = READY
```

含义是：

```text
这条脚本真实可拍，但该内容路线的商业效果仍需测试。
```

# 11. 数据反馈与归因

发布反馈不能直接改写 Script Director 规则。

低播放可能来自：

- 账号权重；
- 发布时间；
- 达人影响力；
- 投流；
- 商品价格；
- 店铺信任；
- 首帧；
- 拍摄执行；
- 剪辑；
- 商品本身；
- 脚本方向；
- Hook；
- Product Proof；
- 样本量不足。

需要按问题归因：

| 问题 | 优先更新模块 |
|---|---|
| 产品不适合当前内容路线 | SD-0 |
| 内容组合错误 | CP-0 |
| 构想本身无价值 | IC-0 |
| 主传播命题错误 | SD-1 Strategy |
| Hook 弱 | Hook Knowledge / SD-1 |
| Product Proof 不成立 | Proof Knowledge / 产品测试 |
| 脚本不可拍 | Shot Production / SD-1 |
| 成片偏离脚本 | Production / QA |
| 发布主体不适合 | SD-0 / Publishing |
| 数据无明确原因 | 继续收集，不急于改规则 |

# 14. 当前 Custom GPT 范围验收测试

未来审核项目时，至少执行以下测试。

预期：

```text
正常生成 READY / PROVISIONAL / BLOCKED 和正式脚本。
```

## Case 2｜要求做市场研究

用户：

```text
帮我判断这个商品在 TikTok 是否值得做。
```

预期：

```text
明确说明超出 Script Director 范围。
```

## Case 3｜要求决定达人还是产品路线

用户：

```text
这个品应该找达人，还是自己拍产品视频？
```

预期：

```text
不进行路线决策，要求用户先完成上游决策。
```

## Case 4｜没有构想

用户只提供商品参数。

预期：

```text

不得自行研究市场后替用户创建完整路线。
```

## Case 5｜给定构想但证据不足

预期：

```text
输出 PROVISIONAL 或 BLOCKED。
不得改成市场分析任务。
```

## Case 6｜给定构想且路线可能不合理

即使模型认为上游路线可能有问题，也只能：

```text

但仍按照输入事实判断脚本是否可写。
```

不得擅自替用户改成达人路线。

# 17. 冻结结论

## 17.1 完整系统

长期完整系统应包含：

```text
方向决策
+
内容组合
+
构想
+
剧本
+
生产
+

+
发布反馈
```

## 17.2 当前 Custom GPT

当前 Custom GPT 永久保持：

> 只把已经确定的一条视频构想转换成正式拍摄脚本。

它不承担：

```text
市场判断
选品
内容路线判断
达人策略
账号策略
产品内容组合
发布策略
数据预测
```

## 17.3 实现原则

```text
完整系统可以扩展
单个 Agent 必须收敛
```

未来新增功能时，优先新增独立模块和清晰接口，而不是继续扩大 Script Director 的职责。
